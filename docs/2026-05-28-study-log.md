## 다중 이미지 핵심 개념

이미지 로직에서 가장 중요한 점은 기존 서버 이미지와 새로 첨부한 이미지가 같은 타입의 데이터가 아니라는 것이다.

```txt
기존 서버 이미지
-> 이미 서버에 올라가 있음
-> File 객체가 아님
-> string URL만 있음

새로 첨부한 이미지
-> 브라우저에서 선택한 File 객체
-> 아직 서버에 없음
-> uploadFile을 먼저 해야 함
```

그래서 둘을 화면에서 같이 다루기 위해 `ImagePreview` 타입으로 통합했다.

```ts
type ImagePreview = {
  file?: File;
  previewUrl: string;
  uploadedUrl?: string;
  isExisting: boolean;
  name: string;
};
```

각 필드의 의미는 아래와 같다.

```txt
file
-> 새로 첨부한 이미지의 File 객체
-> 기존 이미지는 File이 아니므로 없음

previewUrl
-> img src에 넣어 화면에 보여줄 주소
-> 화면 미리보기용

uploadedUrl
-> 서버에 이미 올라간 이미지 URL
-> updateTravelproduct의 images 배열에 들어갈 값

isExisting
-> 기존 서버 이미지인지 새 이미지인지 구분

name
-> key 또는 alt에 사용할 이름
```

## 기존 이미지와 새 이미지

수정 페이지에서 기존 상품 이미지는 서버에서 `string[]`으로 온다.

```ts
data.fetchTravelproduct.images
```

이 값은 화면에서 보여주기 위해 아래처럼 `ImagePreview` 형태로 바꾼다.

```ts
{
  previewUrl: `https://storage.googleapis.com/${file}`,
  uploadedUrl: file,
  name: file,
  isExisting: true,
}
```

여기서 중요한 점은 `uploadedUrl`이다.

```txt
uploadedUrl
-> 최종 updateTravelproductInput.images에 다시 넣을 서버 이미지 URL
```

새로 첨부한 이미지는 아래와 같은 형태다.

```ts
{
  file,
  previewUrl: URL.createObjectURL(file),
  isExisting: false,
  name: file.name,
}
```

새 이미지는 `File` 객체가 있으므로 `uploadFile` mutation으로 먼저 업로드해야 한다.

## 수정 시 이미지 전송 흐름

수정 submit에서 이미지 처리는 아래 순서로 이해하면 된다.

```txt
1. imageFiles에서 isExisting === true인 기존 이미지 추출
2. 기존 이미지의 uploadedUrl만 뽑음
3. imageFiles에서 isExisting === false인 새 이미지 추출
4. 새 이미지는 uploadFile 실행
5. uploadFile 결과에서 url만 뽑음
6. 기존 URL 배열과 새 URL 배열을 합침
7. updateTravelproductInput.images에 최종 string[] 전송
```

최종적으로 서버에 보내야 하는 값은 반드시 아래 형태여야 한다.

```ts
images: ["기존이미지URL", "새업로드이미지URL"]
```

아래 값들은 서버에 보내면 안 된다.

```txt
File 객체
previewUrl
URL.createObjectURL(file) 값
```

`previewUrl`은 화면용이고, `images`는 서버 저장용이다.

## 이미지 삭제

이미지 삭제는 `imageFiles` 배열에서 해당 index를 제거하는 방식이다.

```txt
기존 이미지 삭제
-> imageFiles에서 제거
-> finalImageUrls에 포함되지 않음
-> updateTravelproduct 시 서버 images에서도 빠짐

새 이미지 삭제
-> imageFiles에서 제거
-> uploadFile 대상에서도 빠짐
```

새 이미지의 `previewUrl`은 `URL.createObjectURL`로 만든 임시 URL이므로 삭제할 때 `URL.revokeObjectURL`로 정리할 수 있다.

## tags 처리

input에 입력한 태그 값은 배열이 아니라 문자열이다.

```txt
"제주, 호텔, 바다"
```

서버의 `tags`는 `string[]`을 기대하므로 submit 전에 배열로 변환해야 한다.

```ts
const tags = data.tags
  ? data.tags
      .split(",")
      .map((tag) => tag.trim())
      .filter(Boolean)
  : [];
```

흐름은 아래와 같다.

```txt
"제주, 호텔, , 바다,"

split(",")
-> ["제주", " 호텔", " ", " 바다", ""]

trim()
-> ["제주", "호텔", "", "바다", ""]

filter(Boolean)
-> ["제주", "호텔", "바다"]
```

처음에는 `tags: data.tags ? [data.tags] : []`처럼 보냈다.

이 경우 결과는 아래처럼 된다.

```ts
["제주, 호텔, 바다"]
```

배열은 맞지만 태그 3개가 아니라 문자열 1개짜리 배열이다.

원하는 결과는 아래다.

```ts
["제주", "호텔", "바다"]
```

## filter(Boolean)

`filter(Boolean)`은 배열에서 falsy 값을 제거하는 패턴이다.

제거되는 값은 아래와 같다.

```txt
false
0
""
null
undefined
NaN
```

문자열 배열에서 빈 문자열을 제거할 때 자주 쓴다.

```ts
["제주", "", "호텔"].filter(Boolean);
// ["제주", "호텔"]
```

주의할 점도 있다.

```ts
[0, 1, 2].filter(Boolean);
// [1, 2]
```

`0`이 의미 있는 값이면 `filter(Boolean)`을 쓰면 안 된다.

## 타입가드

이미지 업로드 결과에서 URL만 추출할 때 아래 패턴을 사용했다.

```ts
.filter((url): url is string => Boolean(url));
```

이 코드는 두 가지 일을 한다.

```txt
1. 런타임에서 undefined, null, "" 같은 값을 제거
2. TypeScript에게 이 배열은 string[]이라고 알려줌
```

`result.data?.uploadFile.url`은 `?.` 때문에 `string | undefined`가 될 수 있다.

그래서 그냥 `filter(Boolean)`만 쓰면 실행 결과는 괜찮아도 TypeScript가 여전히 `(string | undefined)[]`로 볼 수 있다.

타입가드를 쓰면 TypeScript가 최종 타입을 `string[]`으로 이해한다.

```ts
const imageUrls = resultImage
  .map((result) => result.data?.uploadFile.url)
  .filter((url): url is string => Boolean(url));
```

## react-hook-form과 Controller

Ant Design `Input`, `InputNumber`, `ReactQuill` 같은 컴포넌트는 `Controller`로 연결했다.

중요한 점은 `field.onChange`를 반드시 호출해야 한다는 것이다.

```tsx
onChange={(event) => {
  field.onChange(event);
  handleGetTags(event);
}}
```

`field.onChange(event)`는 react-hook-form에게 값이 바뀌었다고 알려준다.

잘못된 예시는 아래와 같다.

```tsx
onChange={console.log(field.onChange)}
```

이 코드는 입력할 때 실행되는 것이 아니라 렌더링 시점에 바로 실행된다.

그리고 `console.log`의 반환값은 `undefined`이므로 `onChange`에 함수가 아니라 `undefined`가 들어가 타입 에러가 발생한다.

입력값을 확인하려면 아래처럼 이벤트 함수 안에서 찍어야 한다.

```tsx
onChange={(event) => {
  console.log(event.target.value);
  field.onChange(event);
}}
```

## accessToken / refreshToken

오늘 인증 흐름도 함께 정리했다.

```txt
accessToken
-> API 요청할 때 Authorization 헤더에 붙이는 짧은 수명의 토큰
-> 현재 코드에서는 Zustand 메모리에 저장

refreshToken
-> accessToken이 만료됐을 때 새 accessToken을 받기 위한 긴 수명의 토큰
-> 현재 코드에서는 직접 변수로 저장하지 않음
-> 백엔드가 쿠키에 저장
```

refreshToken은 코드에서 직접 보이지 않는다.

대신 요청에 쿠키를 포함시키기 위해 아래 옵션이 중요하다.

```ts
credentials: "include"
```

이 옵션은 `uploadLink`와 `getAccessToken`의 `GraphQLClient`에 들어가야 한다.

## 로그인 흐름

로그인 페이지에서는 `loginUser` mutation을 호출한다.

```txt
loginUser 성공
-> accessToken 응답 받음
-> setAccessToken으로 Zustand에 저장

```

코드상으로는 accessToken만 직접 다룬다.

```ts

```

## Zustand accessToken store

`src/commons/stores/accessToken.ts`는 accessToken을 전역 상태로 보관한다.

```ts
export const useAccessTokenStore = create<IUseAccessTokenStore>((set) => ({
  accessToken: "",
  setAccessToken(newAccessToken: string) {
    return set(() => {
      return { accessToken: newAccessToken };
    });
  },
}));
```

컴포넌트 안에서는 hook처럼 사용한다.

```ts
const { setAccessToken } = useAccessTokenStore();
```

React 컴포넌트 바깥에서는 hook 호출이 아니라 `getState()` / `setState()`를 사용한다.

```ts
useAccessTokenStore.getState().accessToken;
useAccessTokenStore.setState({ accessToken: newAccessToken });
```

Apollo Link는 React 컴포넌트가 아니므로 `useAccessTokenStore()`처럼 훅 호출하면 안 된다.

## Apollo authLink

`authLink`는 GraphQL 요청이 나가기 직전에 accessToken을 Authorization 헤더에 붙이는 역할이다.

```ts
const authLink = new ApolloLink((operation, forward) => {
  const accessToken = useAccessTokenStore.getState().accessToken;

  operation.setContext({
    headers: {
      Authorization: accessToken ? `Bearer ${accessToken}` : "",
    },
  });

  return forward(operation);
});
```

흐름은 아래와 같다.

```txt
GraphQL 요청 발생
-> authLink 실행
-> Zustand에서 accessToken 읽음
-> Authorization 헤더 추가
-> forward(operation)으로 다음 link에 넘김
```

## Apollo errorLink와 refresh

`errorLink`는 GraphQL 요청 실패를 중간에서 잡는다.

```ts
const errorLink = onError(({ graphQLErrors, operation, forward }) => {
  ...
});
```

처음에는 `UNAUTHENTICATED` 에러 코드를 기대했지만, 실제 과제 서버는 아래처럼 내려줬다.

```txt
extensions.code: INTERNAL_SERVER_ERROR
message: 회원정보 인증에 실패하였습니다.
```

그래서 현재 조건은 메시지도 함께 확인한다.

```ts
err.message === "회원정보 인증에 실패하였습니다."
```

인증 실패를 감지하면 아래 흐름이 실행된다.

```txt
1. getAccessToken으로 새 accessToken 요청
2. 새 accessToken이 없으면 로그인 페이지 이동
3. 새 accessToken이 있으면 Zustand에 저장
4. 실패했던 요청의 Authorization 헤더 교체
5. forward(operation)으로 실패했던 요청 재시도
```

## getAccessToken

`getAccessToken`은 refreshToken을 새로 발급받는 함수가 아니다.

정확히는 아래 역할이다.

```txt
쿠키에 이미 들어있는 refreshToken으로 새 accessToken을 재발급받는 함수
```

```ts
const graphQLClient = new GraphQLClient(
  "https://main-practice.codebootcamp.co.kr/graphql",
  {
    credentials: "include",
  },
);
```

`credentials: "include"`가 있어야 쿠키의 refreshToken이 서버로 같이 간다.

```graphql
mutation restoreAccessToken {
  restoreAccessToken {
    accessToken
  }
}
```

서버가 새 accessToken을 주면 반환한다.

```ts
return newAccessToken;
```

## fromPromise와 forward

`getAccessToken()`은 Promise를 반환한다.

Apollo Link 흐름에서 Promise 결과를 이어 쓰기 위해 `fromPromise`를 사용한다.

```ts
return fromPromise(
  getAccessToken().then((newAccessToken) => {
    ...
  }),
).flatMap(() => forward(operation));
```

역할은 아래처럼 이해하면 된다.

```txt
fromPromise
-> accessToken 재발급 Promise를 Apollo Link 흐름으로 변환

forward(operation)
-> 방금 실패했던 GraphQL 요청을 다시 보냄
```

새 accessToken을 받은 뒤에는 실패했던 요청의 헤더를 바꾼다.

```ts
operation.setContext({
  headers: {
    ...operation.getContext().headers,
    Authorization: `Bearer ${newAccessToken ?? ""}`,
  },
});
```

이후 `forward(operation)`으로 재시도한다.

## throw new Error

새 accessToken을 받지 못하면 아래 코드가 실행된다.

```ts
throw new Error("No access token");
```

`return`은 현재 콜백만 조용히 끝낼 수 있다.

반면 `throw`는 Promise 흐름을 실패로 바꾸고, 뒤의 재시도가 진행되지 않게 한다.

즉 아래 상황을 막기 위한 코드다.

```txt
새 accessToken 없음
-> 그런데 실패했던 요청을 다시 보냄
-> 또 인증 실패
```

## Apollo 버전 차이

현재 프로젝트는 `@apollo/client` 3 계열이다.

```txt
@apollo/client: ^3.11.8
```

그래서 `onError(({ graphQLErrors, operation, forward }) => {})` 방식이 맞다.

Apollo Client 4에서는 에러 처리 방식이 달라져 `CombinedGraphQLErrors` 같은 형태를 확인해야 한다.

## 오늘의 결론

오늘 배운 내용은 양이 많았다.

특히 아래 두 축이 핵심이다.

```txt
이미지
-> 기존 서버 URL과 새 File을 구분해서 최종 string[]으로 보낸다.

인증
-> accessToken은 요청용, refreshToken은 재발급용이며 실패한 요청은 새 토큰으로 재시도한다.
```

오늘 목표는 모든 코드를 외우는 것이 아니라, 나중에 다시 봤을 때 어디서부터 확인해야 하는지 기준점을 만드는 것이다.

지금 단계에서는 아래 정도를 기억하면 충분하다.

```txt
기존 이미지는 File이 아니다.
새 이미지만 uploadFile 한다.
최종 images에는 string URL만 들어간다.

accessToken은 Zustand에 있고 refreshToken은 쿠키에 있다.
authLink는 토큰을 붙이고 errorLink는 토큰 만료를 복구한다.
```
