# 2026-05-22 스터디 일지

## 작업 범위

오늘은 `homework19`의 나만의 콘텐츠 목록 과제를 이어서 진행했다. 주제는 여행 위시리스트로 유지했고, Firebase Firestore를 사용해 목록 조회, 등록, 상세 조회, 수정 흐름을 실제 페이지 구조에 맞춰 구현했다. TypeScript와 Firestore의 데이터 타입이 아직 익숙하지 않아 중간중간 버벅였지만, 게시판 CRUD의 큰 흐름은 훨씬 선명해졌다.

## 진행한 작업

### 1. Firestore 데이터 구조 재정리

- 기존 `travelWishlists` 컬렉션을 삭제한 뒤 새로 테스트 데이터를 생성했다.
- 데이터 기본 형태는 기존 인터페이스를 유지했다.

```ts
interface IFetchData {
  id: string;
  place: string;
  country: string;
  memo?: string;
  season: string;
  priority: string;
}
```

- 최신순 정렬을 위해 `createdAt` 필드를 추가했다.
- `createdAt`은 Firestore의 `Timestamp` 타입으로 저장했다.
- 여행지 이름, 국가, 계절, 우선순위, 메모를 실제 여행 위시리스트처럼 의미 있는 값으로 채운 테스트 데이터 84개를 생성했다.
- 최신순 조회에는 문서 id가 아니라 `createdAt` 기준 정렬이 더 자연스럽다는 점을 확인했다.

### 2. 목록 조회 구현

- `src/app/myapis/page.tsx`는 직접 UI를 작성하지 않고 `MyApisList`와 `useMyApisList`를 조립하는 역할로 정리했다.
- `src/components/myapis-list/hook.ts`에서 Firestore 목록 조회 로직을 관리했다.
- `getDocs`, `collection`, `query`, `orderBy("createdAt", "desc")`를 사용해 여행 위시리스트 목록을 최신순으로 가져왔다.
- Firestore의 `doc.data()`에는 문서 id가 포함되지 않으므로, 목록 데이터에 `id: doc.id`를 직접 붙였다.

```ts
const datas = result.docs.map((el) => ({
  id: el.id,
  ...(el.data() as Omit<IFetchData, "id">),
}));
```

- 목록 테이블에서 각 행을 클릭하면 해당 문서 id를 사용해 `/myapis/[myapiId]` 상세 페이지로 이동하도록 만들었다.
- 처음에는 `data[0].id`처럼 직접 접근하려고 하면서 초기 렌더링 시 빈 배열 문제를 겪었고, 이후 `data.map()` 안에서 클릭한 `el.id`를 넘기는 방식이 더 안전하다는 점을 확인했다.

### 3. 등록 페이지 구현

- `src/app/myapis/new/page.tsx`는 `MyApisWrite`와 `useMyApisWrite({ isEdit: false })`를 조립하도록 정리했다.
- 등록/수정에서 같은 폼 컴포넌트를 재사용하기 위해 `isEdit` 값을 넘기는 구조를 만들었다.
- `inputState` 하나로 `place`, `country`, `memo`, `season`, `priority` 값을 관리했다.
- 일반 input은 `event.target.name`을 활용해 하나의 `handleInputChange`로 처리했다.
- Ant Design `Select`는 일반 input 이벤트와 구조가 다르므로 `handleSelectChange("season", value)`처럼 필드 이름을 직접 넘기는 방식으로 처리했다.
- 필수값인 여행지 이름과 국가/지역이 비어 있으면 `Modal.warning`을 보여주고 등록을 막았다.
- 등록 시에는 `addDoc`을 사용하고, `createdAt: serverTimestamp()`를 함께 저장했다.

### 4. 상세 조회 구현

- `src/app/myapis/[myapiId]/page.tsx`는 `MyApisDetail`과 `useMyApisDetail`을 조립하는 구조로 정리했다.
- 상세 조회에서는 `collection + getDocs`가 아니라 `doc + getDoc`을 사용해야 한다는 점을 확인했다.

```ts
const travelWishlists = doc(db, "travelWishlists", String(params.myapiId));
const result = await getDoc(travelWishlists);
```

- 문서가 없을 수도 있으므로 `result.exists()`로 존재 여부를 확인했다.
- 상세 데이터도 `result.data()`에는 id가 포함되지 않으므로 `result.id`를 직접 합쳤다.
- 상세 화면에서 `place`, `country`, `memo`, `season`, `priority`를 보여주고, 목록/수정 페이지 이동 버튼을 연결했다.

### 5. 수정 페이지 구현

- `src/app/myapis/[myapiId]/edit/page.tsx`에서 `useMyApisWrite({ isEdit: true })`를 사용하도록 만들었다.
- 수정 페이지에 들어오면 `getDoc`으로 기존 문서를 불러오고, 그 값을 `inputState`에 넣어 폼 초기값으로 사용했다.

```ts
const data = result.data() as ITravelWishInput;

setInputState({
  ...data,
});
```

- 수정 버튼을 누르면 `addDoc`이 아니라 `updateDoc`으로 기존 문서를 수정하도록 분리했다.
- 등록과 수정의 차이를 다시 정리했다.

```txt
등록 = 새 문서를 만드는 것 = addDoc
수정 = 기존 문서를 바꾸는 것 = updateDoc
```

- 등록 페이지에서는 URL에 `myapiId`가 없기 때문에 `params.myapiId`가 `undefined`가 되는 이유도 확인했다.
- 수정 페이지는 `/myapis/[myapiId]/edit` 경로이므로 `params.myapiId`가 있고, 수정 후 상세 페이지로 돌아갈 때 이 값을 사용할 수 있다.

## 오늘 확인한 개념

### 1. Firestore는 id와 data를 분리한다

Firestore 문서는 id와 데이터가 분리되어 있다.

```txt
doc.id = 문서 id
doc.data() = 문서 안에 저장한 필드들
```

그래서 목록과 상세에서 항상 아래 흐름을 반복하게 된다.

```ts
{
  id: doc.id,
  ...doc.data(),
}
```

GraphQL에서는 `_id`가 응답 데이터 안에 들어오는 경우가 많았지만, Firestore에서는 문서 id를 직접 합쳐야 한다.

### 2. 목록과 상세의 조회 함수가 다르다

Firestore에서는 여러 문서 조회와 단일 문서 조회가 명확히 나뉜다.

```txt
목록 조회 = getDocs(collection 또는 query)
상세 조회 = getDoc(doc)
```

처음에는 `collection(db, "travelWishlists", id)`처럼 접근하려고 했지만, 특정 문서 하나를 가져올 때는 `doc(db, "travelWishlists", id)`가 맞다.

### 3. 최신순 정렬은 `createdAt`이 자연스럽다

Firestore 자동 문서 id는 생성순이 아니므로 최신순 정렬 기준으로 쓰기 어렵다.

```ts
orderBy("createdAt", "desc")
```

따라서 등록 시점에 `createdAt`을 저장하고, 목록 조회에서 그 값을 기준으로 정렬하는 방식이 더 명확하다.

### 4. TypeScript는 아직 어렵지만 역할이 보이기 시작했다

오늘 특히 어려웠던 부분은 Firestore의 `result.data()` 타입이었다. Firestore는 `data()`의 구체적인 필드 구조를 TypeScript에게 자동으로 알려주지 않는다. 그래서 직접 인터페이스를 만들고 타입을 알려줘야 했다.

```ts
interface ITravelWishInput {
  place: string;
  country: string;
  memo?: string;
  season: string;
  priority: string;
}
```

```ts
const data = result.data() as ITravelWishInput;
```

아직 한 번에 흐름이 눈에 들어오지는 않지만, 왜 타입을 지정해야 하는지, `IFetchData[]`와 `ITravelWishInput`이 어떤 역할을 하는지는 조금씩 감이 잡히기 시작했다.

### 5. 페이지네이션은 이번 과제에서 제외하기로 결정

Firestore 페이지네이션은 GraphQL처럼 `page: 3`을 넘기는 방식이 아니라 `startAfter`를 사용하는 cursor 기반 방식이다.

```txt
GraphQL pagination = 3페이지 주세요
Firestore pagination = 이 문서 다음부터 10개 주세요
```

숫자 페이지네이션까지 제대로 구현하려면 각 페이지의 cursor를 관리해야 한다. 이번 과제의 핵심은 Firebase CRUD와 컴포넌트 분리이므로, 페이지네이션은 비용 대비 학습 효율이 낮다고 판단해 제외하기로 했다.

## 오늘 마무리

오늘은 TypeScript와 Firestore가 아직 낯설어서 여러 번 막혔지만, 직접 구현한 결과는 꽤 좋았다. 특히 게시판의 기본 흐름은 많이 잡혔다.

```txt
목록
→ id로 상세 이동
→ 상세에서 수정 이동
→ 수정 페이지에서 기존값 조회
→ updateDoc으로 저장
→ 상세로 복귀
```

이제 게시판 CRUD의 큰 구조는 이해하기 시작했다. 앞으로의 어려움은 게시판 자체보다는 라이브러리별 사용법, 타입 선언, 파일 분리, 데이터 갱신 타이밍 같은 주변 개념이 될 가능성이 크다. 그래도 오늘은 질문을 줄이면서도 스스로 많은 흐름을 연결했고, 다음 과제로 넘어갈 수 있을 만큼 의미 있는 결과를 만들었다.
